 1. SECURITY
═══════════════════════════════════════════════════════════════════════════════

[CRITICAL] S1 — JWT secret has insecure fallback
  File: src/config.js
  Issue: Default value is 'dev-secret-change-me'. If JWT_SECRET env var
         is not set in production, all tokens are signed with a known key.
  Fix:  Fail fast on startup if JWT_SECRET is not provided.

[CRITICAL] S2 — CORS is wide open
  File: src/app.js
  Issue: app.use(cors()) accepts requests from ANY origin.
  Fix:  Whitelist only your frontend URL(s) and trusted API consumers.

[CRITICAL] S3 — Content Security Policy (CSP) disabled
  File: src/app.js
  Issue: contentSecurityPolicy: false in Helmet config. No XSS protection.
  Fix:  Define proper CSP directives for your frontend + CDN sources.

[HIGH] S4 — Refresh tokens stored as plaintext in DB
  File: src/services/auth.js
  Issue: Raw JWT strings stored in refresh_tokens table. DB compromise
         exposes all active sessions.
  Fix:  Hash tokens with bcrypt/sha256 before storage.

[HIGH] S5 — No request/access logging
  Issue: No Morgan, Pino, or Winston. Zero visibility for debugging
         or auditing production traffic.
  Fix:  Add structured logging middleware (recommend Pino).

[HIGH] S6 — audit_log table is dead code
  File: db/schema.sql
  Issue: Table exists but no service, repo, or middleware writes to it.
  Fix:  Create audit service + middleware for sensitive operations.

[HIGH] S7 — SSLCommerz callback has no signature verification
  File: src/routes/payments.js
  Issue: Success/fail/cancel callbacks trust req.body blindly.
         Attacker could forge a successful payment callback.
  Fix:  Validate hash/signature per SSLCommerz documentation.

[HIGH] S8 — custom_js in website_settings allows XSS
  Issue: Shop admins can inject arbitrary JS into storefronts.
  Fix:  Sanitize or sandbox custom JS. Consider CSP nonces.

[HIGH] S9 — validateBody middleware exists but never used
  File: (exists but not wired into any route)
  Issue: All routes accept raw req.body with no schema validation.
  Fix:  Wire Zod/Joi validation to every POST/PUT/PATCH route.

[MEDIUM] S10 — No CSRF protection
  Issue: SSLCommerz callbacks use form POST. localStorage tokens
         are vulnerable to XSS. No CSRF tokens anywhere.
  Fix:  Add CSRF tokens for form-based flows.

[MEDIUM] S11 — No input sanitization (XSS prevention)
  Issue: No html-escape or DOMPurify for user-supplied text/HTML
         stored in JSONB fields.
  Fix:  Sanitize all user input before DB storage.

[MEDIUM] S12 — Super admin x-shop-id header not validated
  File: src/middleware/auth.js
  Issue: Any UUID in the header is accepted without checking shop exists.
  Fix:  Validate shop existence for super admin scope switch.

 2. AUTHENTICATION & AUTHORIZATION
═══════════════════════════════════════════════════════════════════════════════

[HIGH] A1 — No email verification flow
  Issue: Users and customers can register with any email. No confirmation
         step. Could lead to fake accounts and spam.
  Fix:  Send verification email with token on registration. Block login
        until verified.

[HIGH] A2 — No password reset / forgot password
  Issue: No endpoint, no token generation, no email. Users who forget
         their password are locked out permanently.
  Fix:  Implement reset token generation + email. ~2 hour task.

[HIGH] A3 — Customer tokens have no refresh mechanism
  File: src/services/customers.js
  Issue: Only issues access tokens (15 min expiry). Customer sessions
         expire with no way to refresh => terrible UX.
  Fix:  Issue refresh tokens for customer auth too.

[HIGH] A4 — Deactivated users can still log in (BUG)
  File: src/services/auth.js
  Issue: is_active field in users table is never checked at login.
  Fix:  Add `if (!user.is_active) throw ...` check in login().

[MEDIUM] A5 — No account lockout / brute force detection
  Issue: Rate limiter (20 req / 15 min) helps but no per-account lockout
         after N failed attempts.
  Fix:  Track failed attempts per email. Lock after 5 failures for 30 min.

[MEDIUM] A6 — Weak password requirements
  Issue: Only checks password.length < 6. No uppercase, number, or special
         character requirements.
  Fix:  Enforce minimum complexity rules.

[MEDIUM] A7 — No password change endpoint for logged-in users
  Issue: Users cannot change their password from the UI.
  Fix:  Add PATCH /auth/password endpoint.

[LOW] A8 — No last_login_at tracking
  Issue: Architecture plan includes it but actual schema & service don't.
  Fix:  Update last_login_at on successful login.

[LOW] A9 — No JWT key rotation
  Issue: Single static secret for all tokens. Architecture plan recommends
         rotating signing keys periodically.
  Fix:  Implement JWKS with key versioning.

[LOW] A10 — No refresh token limit per user
  Issue: Users can accumulate unlimited refresh tokens.
  Fix:  Cap to max 5 per user, clean up expired tokens.

 3. PAYMENTS
═══════════════════════════════════════════════════════════════════════════════

[CRITICAL] P1 — Refund flow is DB-only (FAKE REFUND)
  File: src/services/payments.js → createRefund()
  Issue: Creates a refund record in DB but NEVER calls SSLCommerz refund
         API. Money is not actually returned to customer.
  Fix:  Wire to SSLCommerz refund endpoint or gateway adapter.

[CRITICAL] P2 — Per-shop SSLCommerz credentials NOT used (BUG)
  File: src/services/payments.js
  Issue: shops table has sslcommerz_store_id/pass columns but service
         always uses global config credentials. All shops share one
         merchant account.
  Fix:  Read credentials from shop record before initiating payment.

[HIGH] P3 — Manual payment doesn't update order payment_status (BUG)
  File: src/services/payments.js → createManualPayment()
  Issue: Creates payment record as 'completed' but order's payment_status
         field is never updated to 'paid'.
  Fix:  Update order.payment_status after capturing payment.

[HIGH] P4 — No payment idempotency keys
  Issue: Retries or double-clicks could create duplicate payment records.
  Fix:  Add idempotency key per payment initiation.

[HIGH] P5 — Tax calculation hardcoded to 0
  File: src/services/orders.js → calculateTotals()
  Issue: tax_amount always returns 0. No tax rules engine.
  Fix:  Implement configurable tax rates (per-shop or per-region).

[HIGH] P6 — Shipping cost hardcoded to 0
  Issue: Same as above. No shipping rate calculation.
  Fix:  Implement shipping zones + weight-based or flat-rate shipping.

[HIGH] P7 — No discount/coupon system
  Issue: discount_amount column exists but there's no coupons table,
         no promo code validation, no discount logic.
  Fix:  Create discounts table + coupon validation service.

[MEDIUM] P8 — Currency hardcoded to BDT
  Issue: No multi-currency support. All amounts assume Bangladeshi Taka.
  Fix:  Add currency field to shops + conversion at checkout.

[MEDIUM] P9 — No payment receipt / invoice generation
  Issue: Customers get no receipt or downloadable invoice.
  Fix:  Generate PDF invoices on payment capture.

[MEDIUM] P10 — Only SSLCommerz integrated
  Issue: No Stripe, PayPal, bKash, or other payment gateways.
  Fix:  Implement payment gateway adapter pattern.

[HIGH] P11 — Subscription plan billing not implemented
  Issue: subscription_plans table has pricing columns but no recurring
         charge, plan enforcement, or upgrade/downgrade flow.
  Fix:  Implement billing system or integrate Stripe Billing.

 4. ORDERS
═══════════════════════════════════════════════════════════════════════════════

[CRITICAL] O1 — No stock validation before order creation
  File: src/services/orders.js
  Issue: Orders are created without checking if items are in stock.
         Variant inventory can go NEGATIVE.
  Fix:  Check stock before decrement. Reject order if insufficient.

[CRITICAL] O2 — Product-level stock never decremented (BUG)
  Issue: Only variant inventory is decremented. Products without variants
         have stock_quantity that is NEVER reduced on order.
  Fix:  Decrement product stock_quantity when order doesn't use variants.

[HIGH] O3 — No order cancellation with inventory restore
  Issue: Setting status to 'cancelled' doesn't restock items, void
         payments, or cancel in-flight deliveries.
  Fix:  Implement full cancellation flow with rollback.

[HIGH] O4 — No order state transition validation
  File: src/services/orders.js → updateOrderStatus()
  Issue: Can go from 'delivered' back to 'pending'. No state machine.
  Fix:  Define valid transitions: pending → confirmed → processing →
        shipped → delivered. Cancellation from any pre-delivered state.

[HIGH] O5 — Status list missing 'pending_payment' and 'refunded'
  Issue: Schema CHECK constraint allows these values but service VALID
         array doesn't include them.
  Fix:  Sync service status list with DB constraint.

[MEDIUM] O6 — No human-readable order number
  Issue: Only UUIDs. Architecture plan shows order_number BIGSERIAL.
  Fix:  Add auto-incrementing order number column.

[MEDIUM] O7 — No order confirmation email
  Issue: Nothing sent to customer on order placement or status change.
  Fix:  Send transactional emails via SendGrid/SES.
